CLEAN CODE

"Code is read much more often than it is written."

01

SRP: Single Responsibility

A component or function should have one, and only one, reason to change. If a module handles logic, database, and UI at the same time, it becomes fragile and hard to test.

02

Meaningful Naming

Names should reveal intent. Avoid cryptic variables like "data" or "temp". Clean code acts as its own documentation, making comments almost unnecessary.

03

Eliminate Dead Code

Commented-out code is digital noise. According to your course, "we might need this later" leads to hoarding. Trust your Version Control (Git) to remember the past; keep your present code clean.
