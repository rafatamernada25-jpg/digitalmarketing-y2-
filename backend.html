Extract common functionality into reusable components. The "Rule of Three" suggests refactoring after the third instance of similar code appears to improve maintainability.
Manual Approach (Repetitive)
dog1_name = "Buddy"
dog1_age = 3
dog2_name = "Miles"
dog2_age = 5
Class (Reusable Component)
class Dog:
    def __init__(self, name, age):
        self.name = name
        self.age = age

buddy = Dog("Buddy", 3)
Design each function or class to have only one reason to change (SRP - Single Responsibility Principle). This reduces complexity and makes unit testing significantly easier.
Complex Function
def handle_order(order):
    # Validates AND Saves
    if len(order) > 0:
        db.save(order)
        print("Success")
SRP Refactoring
def validate(order): 
    return len(order) > 0

def save_to_db(order): 
    db.save(order)
Professional standards require eliminating "Magic Numbers". Use named constants to clarify intent and prevent bugs during global modifications.
Magic Numbers
if weight < 5:
    return 4.99
Named Constants
SHIPPING_LIMIT = 5
BASE_RATE = 4.99

if weight < SHIPPING_LIMIT:
    return BASE_RATE

What are Web Services?
Web services enable seamless application-to-application communication over the internet, using standard protocols and formats to ensure interoperability regardless of programming language or platform.
API vs Web Service:
An API (Application Programming Interface) is the interface that allows applications to communicate; a web service is a specific type of API accessed over a network.

Primary Objectives

  • Enable Interoperability: Allow applications built on different platforms (Java, .NET, Python) to exchange data and services seamlessly.
  • Facilitate Communication: Provide a standardized way for systems to interact over a network using common protocols like HTTP/HTTPS.
  • Promote Reusability: Expose existing functionalities as services that can be reused by multiple client applications.
  • Integrate Disparate Systems: Connect internal and third-party applications (payment gateways, mapping services) to create rich solutions.
  • Ensure Scalability: Design systems that handle increased demand by loosely coupling components.
Styles and Formats:
The choice between REST and SOAP, or JSON and XML, defines the rigidity and performance of your system.
JSON Format (Modern)
{
  "orderId": 1,
  "status": "pending",
  "total": 99.99
}
XML Format (Classic)
<order>
  <id>1</id>
  <status>pending</status>
</order>
REST (Representational State Transfer): An architectural style that uses standard HTTP methods (GET, POST, PUT, DELETE). It is the de facto standard for modern web services.

Core REST Principles

  • Statelessness: Each request must contain all the information necessary to be processed. The server stores no client context between requests.
  • Uniform Interface: Consistent use of HTTP verbs to manipulate resources.
  • Loose Coupling: Client and server can evolve independently as long as the interface remains unchanged.
  • HATEOAS: Clients interact with the application through hypermedia links provided dynamically by the server.

URI Design (Golden Rules)

URIs should be based on Nouns (resources) and not on verbs (actions).
❌ BAD DESIGN (Verbs)
GET /getUserDetails
POST /createNewOrder
GET /deleteCustomer?id=5
✅ GOOD DESIGN (Nouns + HTTP Verbs)
GET /users/{id}
POST /orders
DELETE /customers/5
Use plural nouns for collections (/customers) and hierarchical paths for relationships (/customers/5/orders).

Professional Standards & Anti-patterns

  • Avoid "Chatty" APIs: Do not create too many small resources that force the client to make multiple requests for a single action (Chatty I/O).
  • Database Abstraction: The API should NOT mirror your internal database structure. Introduce a mapping layer to isolate the client from schema changes.
  • Pseudo-resources: For non-resource operations (e.g., a calculator), use functional URIs sparingly.
Pseudo-resource Example
GET /calculate/add?op1=10&op2=20 -> 200 OK (Body: 30)
